# 👺 Goblin

Codex의 Responses API(Streaming)를 OpenAI Compatibility API로 변환하는 리버스 프록시 서버입니다.

## 왜 이 프로젝트를 만드나요?

OpenAI의 구독 또는 종량제 대신에 저렴한 오픈소스 모델(e.g. GLM-5.1, Kimi K2.6) 클라우드 제공자(e.g. OpenCode Go, Ollama Cloud, CrofAI)를 Codex에서 사용하기 위해서입니다.
대개 오픈소스 모델 클라우드 제공자를 사용하는 경우에 보통 OpenCode 또는 Pi와 같은 에이전트 하네스를 사용합니다. 
개발 중인 현재 주로 사용하는 에이전트 하네스도 Pi입니다.
하지만 이전에 OpenAI GPT Pro 구독을 사용했을 때의 Codex 앱의 사용 경험을 잊지 못했습니다.
많은 편의 기능이 있는 올인원 앱이었기 때문입니다.

TUI 기반 에이전트 하네스도 이미 충분히 꽤 괜찮고 편리합니다.
하지만 도구 호출 블록 폴딩, 하위 에이전트 진행 상태 및 스레드 내용 보기와 같은 고도화된 제어와 UI를 TUI에서 보기가 점점 불편했습니다.
TUI 특성상 마우스로 제어 가능한 표현의 한계가 있어서 인지성이 크게 떨어집니다.
각 컴포넌트의 텍스트 폰트 크기도 모두 같아서 모든 기능과 상태가 동등한 수준으로만 표현되어 산만한 느낌이 듭니다.
다른 오픈소스 GUI 기반 에이전트 하네스는 Codex 앱을 제외하고는 아직 취미 개발 수준의 안정성과 편의성을 제공했습니다.

그래서 오픈소스 모델 클라우드와 Codex 앱을 같이 쓰기 위해서 이 프로젝트를 만들었습니다.

## 왜 Go로 구현하나요?

주 기능은 리버스 프록시 서버이지만 Codex와 다양한 오픈소스 모델 제공자를 연결해서 사용하도록 하는 것이 목표입니다.
그래서 많은 편의성 기능을 CLI로 구현할것으로 예상되기 때문에 Go를 선택했습니다.

Go는 컴파일 언어이면서 문법이 다른 언어들에 비해 단순하기 때문에 빌드 속도가 꽤 빠릅니다.
C, C++, Rust와 같은 언어는 매크로, 템플릿 등 전처리가 필요한 기능이 있기 때문에 빌드 속도가 느립니다.
Go는 스크립트 언어만큼의 개발 반복 실행 속도는 아니지만 컴파일 언어 중에서는 최고입니다.
그래서 다른 컴파일 언어들에 비해 개발 반복 실행 속도가 빠른 편입니다.
그리고 컴파일 후 바이너리 실행파일이기 때문에 CLI 콜드 스타트가 빠릅니다.

스크립트 언어는 개발 중에 컴파일 단계를 거치지 않기 때문에 개발 속도가 빠르다는 장점이 있습니다.
OpenClaw, Hermes Agent 등 많은 오픈소스 CLI 들이 Node.js 또는 Python 기반입니다.

CLI를 스크립트 언어로 구현할 경우에 CLI 콜드 스타트 지연이 꽤 길고 지루합니다.
CLI 명령을 입력할 때마다 매번 모든 수십 수백개의 스크립트 파일을 로드하고 파싱하고 실행하기 때문입니다.
개발 편의성은 좋지만 CLI를 사용하는 수십 수백명의 사용자가 이 비용을 지불합니다.
작은 프로젝트는 문제없지만 프로젝트의 규모가 커지면서 많은 의존성을 갖게 되면 이 문제를 해결해야되는 시점이 옵니다.
`my-awesome-app --help`를 입력하고 수초 뒤에야 메시지가 보이기 시작합니다.
그러면 의존 스크립트를 초기에 모두 로드하지 않는 동적 로딩 구조로 변경해야 합니다.
Python에서 최상단 import 대신에 특정 함수 내에서 import하는 경우가 그 예입니다.
모듈간 의존성을 파악하고 공유 의존성을 최소화하고 각 CLI 기능에 따라 정말로 필요한 모듈만 import하도록 최적화해야합니다.
동적 로딩 구조이기 때문에 개발 중 특정 기능 로딩의 장애를 인지하기 어려워져서 각 CLI 기능별로 스모크 테스트도 반드시 구성해야 합니다.

한번 실행했을 때 오래 실행되는 서버의 경우 느린 콜드 스타트가 문제는 아닙니다.
하지만 CLI는 자주, 많이, 많은 사람 및 AI 에이전트가 반복적으로 실행하도록 설계된 인터페이스이기 때문에 느린 콜드 스타트는 문제입니다.
이 콜드 스타트 문제를 스크립트 언어 내에서 해결하는 것은 스크립트 언어를 선택했을 때 기대한 그런 즐거운 기능 작업이 아닙니다.
그리고 많은 리소스를 들여서 문제를 해결해봤자 결과물은 컴파일 언어들보다 훨씬 느립니다.

Go를 선택한 또 다른 중요한 이유는 의존성 관리의 보안성입니다.
npm은 패키지 압축 파일의 내용이 바뀌었는지 검증할 방법이 없습니다.
메인테이너 계정이 탈취되면 같은 버전 번호로 악성 압축 파일을 다시 올릴 수 있고, lock 파일은 이걸 걸러내지 못하고 새 해시로 갱신됩니다.
`^1.0.0` 같은 caret 범위에서 최신 버전이 올라갈 때도 마찬가지입니다.

반면 Go는 sum.golang.org라는 투명성 로그에 모듈 버전의 압축 파일 해시를 한번 등록하면 영구히 고정합니다.
이후에 Git 태그를 재작성하거나 같은 버전 번호로 다른 압축 파일을 올려도 해시가 다르면 `checksum mismatch`로 설치가 거절됩니다.
